# CLAUDE.md - 协作规则

## 背景
我在通过逆向 Claude Code 来学习 agent 架构，边学边搭建自己的 agent。

## 开发节奏
- 按 PLAN.md 的 Step 顺序来，一步一步走
- 每完成一个 Step，等我确认理解了再进入下一步
- 不要一口气写完所有代码

## 解释义务
- 每个重要的设计决策，说明**为什么这么做**和**有什么替代方案**
- 关键模块写完后，简要说明它的作用和在整体架构中的位置
- 如果有多种实现方式，列出选项让我选

## 教学风格
- 我不是专业程序员，解释技术概念时**用类比和比喻**，不要用术语解释术语
- 好的例子："ESM 和 CommonJS 就像简体和繁体，功能一样写法不同"
- 坏的例子："ESM 是 ECMAScript Module 规范，支持静态分析和 tree-shaking"
- 先说**它是什么、解决什么问题**，再说技术细节
- 如果一个概念能用 3 句话讲清楚，就不要用 10 句

## 注释风格
关键函数/模块附带简短注释，说明：
- 它解决什么问题
- 对应 Claude Code 源码的哪个部分（如适用）

## 不要做的事
- 不要跳步骤
- 不要"顺手"加计划外的功能
- 不要用我没接触过的模式而不解释
